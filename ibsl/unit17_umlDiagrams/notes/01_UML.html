UML Class Diagram Basics

A UML Class Diagram is an illustration of the relationships among classes in a object-oriented program (OOP) using the Unified Modeling Language (UML). A UML class diagram resembles a flowchart in which classes are portrayed as boxes, each box having three rectangles inside. The top rectangle contains the name of the class; the middle rectangle contains the attributes of the class; the lower rectangle contains the methods, also called operations, of the class.

*Attributes are also called properties, fields, or instance variables.
*Methods are also called operations or functions.

Visibility

In an UML diagram the visibility of an attribute or method is specified by a symbol in front of its name.

Symbol Visibility Description
- private only available to members of the same class
+ public available to any class
# protected available to all subclasses (inheritance)
~ default or package available to any class in same package
(this is selected if no visibility declared)

Class Relationships

There are several different kinds of relationships that exist among classes in an object-oriented program. They are inheritance, association, aggregation, and composition. Aggregation and composition are sub-types of the association relationship.

  • Inheritance
  • Association
    • Aggregation
    • Composition

Inheritance Relationship

An inheritance relationship between two classes is a relationship in which one class inherits or aquires the attributes and methods of another class. A class derived from another class is called a subclass or child class, whereas the class from which a subclass is derived is called a superclass or parent class. The inheritance relationship is an "is a" relationship. A dog is a pet. A mustang is a horse. Red is a color.

Continuing with the Food Truck example below two new classes has been added to the UML Diagram named Food and Drink that are subclasses of the Item class. A white arrow is used to show an inheritance relationship in a UML diagram. The arrow points from a subclass to its inherited superclass.

Association Relationship

The association relationship is a way of describing that a class knows about another class or there exists some type of dependency between the two classes. An example of an association relationship is when one class contains a method that has a parameter of another class. 

public method printBattingAverage(BaseballPlayer player)
{
    double avg = player.getHits() / player.getAtBats();
}

In an UML diagram the associative relationship is denoted by an arrow. The arrow is unidirectional showing which class knows about which class or which class uses which class.

The association relationship can be divided into two categories: aggregration and composition.

Aggregration Relationship

Aggregation is a special form of association in which there is a stronger relationship between two classes.

When an object ‘has-a’ another object, then you have got an aggregation between them. Aggregation is also called a “has-a” relationship. An example of an aggregrative relationship is when one class is an instance variable (field or property) of another class.

In an UML diagram the aggregation relationship is denoted by an arrow with an open diamond on the end. The direction of arrow specifies which object contains which object.

Composition Relationship

A Composition relationship between two classes is a more restrictive form of an aggregative relationship. When an object contains the other object, if the contained object cannot exist without the existence of container object, then it is called composition. Example: A class contains students. A student cannot exist without a class, therefore there exists a composition relationship between class and students. The difference between aggregation and composition is that composition is more restrictive. When there is a composition between two objects, the composed object cannot exist without the other object. This restriction is not there in aggregation. In both aggregation and composition, direction is a must. The direction specifies, which object contains which object.

In this example, a brain cannot exist without a human body. If the body dies so does the brain. The brain is dependent upon the body.

Aggregation vs Composition

Programmers are often confused as to whether a relationship is aggregation or composition. Composition is when one object owns another object while aggregation is when one object uses another object.

Two phrases which come in handy when trying to determine whether an association between two classes is composition or aggregation are "part-of" and "has". If one object is-part-of another object e.g. Engine is part of Car, then the association or relationship between them is composition. On the other hand, if one object just has another object e.g. Car has the driver then it's aggregation.

In Java, a composition association can be created by placing the keyword final in front of the composed object's variable declaration. Final makes an object reference unchangeable, however the object the variable points to can still be modified or changed. An object variable declared final can never refer to another object, but the object it refers to may change state.

Multiplicity

An association relationship between two classes may involve multiply objects, therefore the relationship between the two classes can take on many forms. Some examples include a one-to-one relationship, a zero-to-many relationship, a one-to-many relationship, and a many-to-many relationship. For example a class can have many students. A hotel can have many rooms. Many states can have many cities.

In a UML diagram multiplicity between classes is denoted by placing numbers above the arrow that describes the relationship between two classes.
